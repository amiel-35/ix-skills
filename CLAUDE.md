# Contribution Guide — ix-skills

## 1. Repo structure

| Path | Role |
|---|---|
| `skills/` | Canonical source: one `.md` file per skill + `.mystaffy.json` override files |
| `dist/` | `.skill` artifacts for claude.ai — gitignored, generated by `build.py` |
| `mystaffy-dist/` | mystaffy artifacts — gitignored, generated by `build.py` |
| `.claude-plugin/plugin.json` | Manifest for Claude Code / Cowork |
| `build.py` | Single entry point for packaging skills |
| `requirements.txt` | Python dependencies (`jsonschema`, `pyyaml`) |

---

## 2. Canonical skill format

Each `skills/<id>.md` file starts with a YAML frontmatter block.

### Required fields

| Field | Type | Description |
|---|---|---|
| `id` | string | kebab-case, matches the filename exactly |
| `label` | string | Display name (in French) |
| `version` | string | semver (`x.y.z`) |
| `description_fr` | string | Triggers + what the skill does / does not do (in French) |
| `description_en` | string | Same in English |
| `icon` | string | Emoji or unicode character |
| `domain` | string | Domain tag — see section 6 |
| `input_types` | string[] | Accepted input types |
| `output_types` | string[] | Produced output types |
| `compatible` | string[] | Always `[claude-ai, claude-code, cowork, gpt, gemini, mystaffy]` |

### Optional fields

| Field | Type | Description |
|---|---|---|
| `category` | string | mystaffy atom type: `atome`, `research`, `critique`, `decision`, `production`, … |
| `aliases` | string[] | Alternative names for the skill |

### Body

The body (everything after the frontmatter) is written **entirely in English**. It describes the role, rules, method, constraints, and output format of the skill.

---

## 3. Naming rules

- `id`: lowercase, kebab-case, matches the `.md` filename **exactly**
- No spaces, no accented characters in ids
- Breaking change → bump **major** version
- New feature → bump **minor** version
- Bug fix → bump **patch** version

---

## 4. Checklist — adding a new skill

1. Create `skills/<id>.md` with complete canonical frontmatter
2. Write the body in English
3. Create `skills/<id>.mystaffy.json` with `category`, `params`, `ui` (minimum)
4. Test: `python3 build.py <id> --validate`
5. Build: `python3 build.py <id>`
6. Verify the skill appears in `.claude-plugin/plugin.json` (update if needed)

---

## 5. Checklist — adding a new domain

1. Choose a `domain` slug (lowercase, from the list in section 6)
2. Add the mapping to the `DOMAIN_TO_METIER` table in `build.py`
3. Document the domain in section 6 of this file

## 7. Packaging targets

| target | description |
|---|---|
| `claude-ai` | `dist/*.skill` for claude.ai |
| `cowork` | validates `.claude-plugin/plugin.json` |
| `mystaffy` | `mystaffy-dist/<id>/` |
| `ix-memory` | `mystaffy-dist/<id>/` — native ix-memory format (manifest.json + skill.md) |
| `codex` | `codex-dist/<id>/SKILL.md` — Codex CLI format (name/description/triggers frontmatter) |

```bash
python3 build.py                              # all targets, all skills
python3 build.py <id>                         # specific skill, all targets
python3 build.py --target claude-ai           # dist/*.skill for claude.ai
python3 build.py --target cowork              # validate .claude-plugin/plugin.json
python3 build.py --target mystaffy            # mystaffy-dist/<id>/
python3 build.py --target mystaffy --domain cognitif
python3 build.py --target ix-memory           # mystaffy-dist/<id>/ — ix-memory format
python3 build.py --target codex               # codex-dist/<id>/SKILL.md — Codex CLI format
python3 build.py --validate                   # validate without writing
```

### Installing into Codex

```bash
bash scripts/install-codex.sh                 # build + install all skills
bash scripts/install-codex.sh <id>            # build + install one skill
bash scripts/install-codex.sh --dry-run       # preview without writing
```

Skills are installed to `~/.codex/skills/<id>/SKILL.md`. Restart Codex to activate.

### Extending build.py for a new target

1. Add the value to the `choices` of the `--target` argument
2. Create a `build_<target>(skill_id, ...)` function in `build.py`
3. Call it from `main()` based on the selected target

---

## 8. Contribution rules

- Skill bodies are **in English**
- `label`, `description_fr`, `description_en` and any `README.md` are **in French** (or bilingual)
- One skill = one source file in `skills/`
- No business logic in skills — skills are prompts, not code
- No external dependencies in `build.py` except `jsonschema` and `pyyaml`
- `mystaffy-dist/`, `dist/`, and `codex-dist/` are gitignored — never commit them

---

## 9. What we do NOT do

- No conditional logic, API calls, or side effects in skills
- No external dependencies in `build.py` (except `jsonschema` and `pyyaml`)
- No generated files in the repo (`dist/`, `mystaffy-dist/`, `codex-dist/` are gitignored)
- No skill ids with accents or uppercase letters
